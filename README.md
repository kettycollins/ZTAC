# Легковагова система контролю доступу Zero Trust для шкільної IT-інфраструктури

Цей репозиторій містить мінімально життєздатний продукт (MVP) системи контекстно-залежного керування доступом (Context-Aware Access Control), побудований на принципах архітектури **Zero Trust («ніколи не довіряй, завжди перевіряй»)**.

Прототип розроблено в межах магістерської дисертації з кібербезпеки. Система динамічно вираховує рівень довіри (**Trust Score**) на основі аналізу трьох векторів контексту: автентичності користувача, типу його пристрою та безпеки поточної мережі.

---

## 🚀 Ключовий функціонал системи (v5.5 MVP)

### 1. Контекстна автентифікація (Context-Aware Authentication)

Система реалізує динамічну оцінку безпеки сесії на основі аналізу чотирьох незалежних векторів телеметрії в режимі реального часу:

- **Ідентифікація та рольова модель (Identity/RBAC):** Диференціація прав для ролей `Admin`, `Teacher`, `Student` та `Guest`.
- **Статус пристрою (Device Trust):** Визначення приналежності девайса — корпоративний керований (`Managed Device`) або власний (`BYOD Device`).
- **Мережевий контекст (Network Trust):** Аналіз локації підключення — внутрішній захищений периметр (`School Network`), домашня мережа (`Home Network`) або публічна зона (`Public Network`).
- **Криптографічний захист каналу (VPN State):** Інтерактивний контроль наявності активного шифрованого тунелю.

### 2. Динамічний аналіз ризиків (Trust Scoring Engine)

Замість застарілого бінарного рішення (так/ні), інтегрований рушій політик (**Policy Decision Point — PDP**) вираховує гнучку математичну оцінку довіри за шкалою від **0 до 100 балів**:

- Базові бали пристрою: `Managed` = 40 балів, `BYOD` = 20 балів.
- Базові бали мережі: `School` = 50 балів, `Home` = 25 балів, `Public` = 10 балів.
- Компенсуючий VPN-бонус: Активація безпечного шифрованого тунелю додає **+25 балів** до загального рейтингу довіри (для легітимних корпоративних сесій).

### 3. Чотирирівнева модель градації ризиків (Trust Levels)

На основі підсумкового `Trust Score` система автоматично класифікує сесію за чотирма рівнями безпеки, що визначають поведінку інтерфейсу:

- **80–100 балів (`Trusted`):** Мінімальний ризик. Надається повний доступ (`FULL`) до об'єктів інфраструктури відповідно до рольової привілегійності.
- **60–79 балів (`Low Risk`):** Незначний ризик (наприклад, робота з дому з VPN). Дозволено безпечну взаємодію з більшістю вузлів.
- **40–59 балів (`Medium Risk`):** Помірний ризик (наприклад, BYOD-пристрій у домашній мережі). Система активує захисні обмеження доступу (`LIMITED` / `READ_ONLY`).
- **0–39 балів (`High Risk`):** Критичний ризик або компрометація контексту. На суворому логічному рівні спрацьовує вердикт `DENY` (тотальна заборона доступу).

### 4. Ізоляція гостьового контуру та захист периметра

Впроваджено суворе залізне правило для неавторизованих або гостьових сесій:

- Доступ для ролі `Guest` дозволено **виключно** за умови фізичного перебування всередині `School Network` (незалежно від типу девайса).
- Будь-яка спроба гостьового підключення із зовнішнього середовища (`Home` або `Public` мережі) відсікається на першому етапі валідації із вердиктом `DENY`.
- На рівні фронтенду (Context-Aware UI) для гостьової ролі повністю деактивується та блокується можливість увімкнення VPN-тоґлера, що виключає несанкціоновану видачу сертифікатів.

### 5. Контекстно-залежна матриця доступу (Granular Authorization)

У разі успішної авторизації користувач потрапляє на динамічну карту об'єктів інфраструктури, де права доступу до кожної окремої картки (Адмін-панель, Журнал оцінок, Профіль студента, Бібліотека) регулюються гранульованими маркерами безпеки:

- `FULL` — повне керування об'єктом.
- `LIMITED` / `READ_ONLY` — режим аудиту (користувач бачить інформацію, але кнопки модифікації даних та внесення змін блокуються на рівні інтерфейсу).
- `DENY` — картка ресурсу візуально гасне (стає напівпрозорою) і блокується для кліків.

### 6. Захист даних від OWASP-вразливостей

- **Нівелювання SQL-ін'єкцій:** Взаємодія з базою даних SQLite в модулі `authenticate.py` реалізована виключно через безпечні параметризовані SQL-запити з підстановкою маркерів `?`. Пряма конкатенація рядків введення повністю заборонена.
- **Безпека веб-інтерфейсу:** Маршрутизація та контроль сесій захищені від спроб прямого обходу форми входу (Broken Access Control) за допомогою серверної перевірки токенів авторизації Flask Session.

### 7. Локалізований моніторинг та SIEM-аудит подій

- **Clean Text Logging:** Модуль реєстрації подій безпеки `logging_utils.py` фіксує кожен крок у JSON-журнал `access_logs.json` у строгому текстовому інженерному форматі без графічних емодзі.
- **Аналітика аномалій:** Система автоматично маркує компрометації паролів, спроби ескалації привілеїв адміністратора або порушення гостьового контуру прапорцем `suspicious_flag: true`, миттєво виводячи інциденти на SIEM-панель адміністратора із ніжно-червоним підсвічуванням рядків.
- **Централізований переклад вердиктів:** Усі текстові причини відмови (PDP Reasons) централізовано обробляються словником `translations.py`, забезпечуючи бездоганне динамічне перемикання мов (`UA/EN`) навіть на екрані жорсткого блокування (`denied.html`), не викликаючи скидання поточної сесії.

## 📁 Структура проекту

- `app.py` — Головний контролер додатка (Flask-маршрутизація, керування сесіями та захист роутів від Broken Access Control).
- `config.py` — Клас централізованої конфігурації шляхів та секретних ключів.
- `database.py` — Ініціалізація бази даних SQLite та безпечне підключення.
- `authenticate.py` — Модуль перевірки облікових даних із захистом від ін'єкцій.
- `policies.py` — Рушій оцінки політик безпеки та розрахунку Trust Score.
- `logging_utils.py` — Модуль безпекового аудиту та виявлення аномальних інцидентів.
- `data/users.db` — Локальна база даних користувачів.
- `logs/access_logs.json` — Файл логів у форматі JSON для реєстрації подій безпеки.
- `templates/` & `static/` — HTML/CSS файли адаптивного інтерфейсу системи.
- `translations.py` - Модуль багатомовності (`UA/EN`).

## 🛠️ Інструкція з розгортання та запуску (Deployment)

Для запуску системи на локальному комп'ютері переконайтеся, що у вас встановлено Python 3.8 або вище.

### 1. Клонування репозиторію

```bash
git clone [https://github.com/kettycollins/ZTAC.git](https://github.com/kettycollins/ZTAC.git)
cd zdac

```

## 2. Встановлення залежностей

Для роботи веб-інтерфейсу потрібен мікрофреймворк Flask. Встановіть його за допомогою pip:

```bash
pip install Flask
```

### 3. Запуск Zero Trust веб-сервера

Запустіть головний файл додатка:

```bash
python app.py
```

Після запуску система автоматично перевірить інфраструктуру, створить базу даних data/users.db та лог-файл logs/access_logs.json. Веб-інтерфейс буде доступний за адресою: http://127.0.0.1:5000/.

### 4. Тестові облікові записи для верифікації

| **Викладач**      | `teacher1`       | `teacher123`      | Гнучкий контекст. З власного пристрою (BYOD) або з   |
|                   |                  |                   | дому система автоматично перемикає картки оцінювання |
|                   |                  |                   | політик безпеки в режим LIMITED або READ_ONLY.       |
| **Студент**       | `student1`       | `student123`      | Надається доступ до Student Hub з дому чи школи.     |
